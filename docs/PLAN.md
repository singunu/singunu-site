# singunu.com — "AI 신건우에게 물어보세요" 웹사이트 구축 계획

## Context

신건우의 개인 AI Q&A 웹사이트. 방문자(친구·채용담당자)가 AI에게 "신건우에 대한 질문"을 하면 `me/` 옵시디언 볼트 기반 페르소나로 답변한다. 답하기 어려운 질문은 연락처를 받아 DB 저장 + 본인 메일 알림. 목적은 ① 인스타 프로필 링크용 재미 서비스 ② 모던 스택 실전 구현·운영 증명 포트폴리오.

**확정된 방향** (사용자 선택):
- 스택: **하이브리드** — Next.js 프론트(Vercel) + Spring Boot API(AWS 경량 서버) + MongoDB Atlas 무료티어 + GitHub Actions
- LLM: **Claude API** (온프레미스 제외 — 트래픽 규모상 API가 월 수천원 이내로 압도적 저렴)
- 목소리 기능: **2차 릴리스** (1차는 텍스트 챗, 아키텍처에 자리만 확보)
- 역할 분담: **코드는 Claude Code, 본인은 운영**(계정·키 발급, DNS, 콘텐츠 검수, 배포 승인)

## 아키텍처

```
[방문자] ── singunu.com (가비아 DNS)
   │
   ▼
Vercel — Next.js 14+ (App Router, TypeScript, Tailwind, Framer Motion)
   │  api.singunu.com (HTTPS)
   ▼
Spring Boot 3 (Java 21, Docker) — EC2 t4g.small or Lightsail $5
   ├─ POST /api/chat        : 가드레일 → RAG(페르소나 문서) → Claude API → 출력 검열 → SSE 스트리밍
   ├─ POST /api/fallback    : 미답변 질문 + 대화이력 + 연락처(자유텍스트) → MongoDB 저장 + 메일 발송
   ├─ Rate Limit (IP당) + 일일 비용 상한 서킷브레이커
   ▼
MongoDB Atlas M0 (무료) — conversations / fallback_questions 컬렉션
Claude API (Haiku 3.5 기본, 필요시 Sonnet)
메일: Gmail SMTP(앱 비밀번호) → 트래픽 늘면 AWS SES 전환
```

CI/CD: GitHub 모노레포(`frontend/` `backend/` `infra/` `persona/`) → GitHub Actions
- 프론트: push → Vercel 자동 배포
- 백엔드: push → 테스트 → Docker 빌드 → GHCR push → EC2 SSH 배포 (Jenkins 제거 — 솔로 프로젝트에 과잉, GitHub Actions가 더 "요즘" 스택)

## 핵심 설계 결정 (원안 대비 변경점과 이유)

| 원안 | 변경 | 이유 |

| 몽고DB | **MongoDB Atlas M0 유지** | 대화 로그는 스키마 유연성이 맞음. 자체 EC2 몽고 운영보다 Atlas가 안전·무료 |
| 온프레미스 LLM | **제외** | GPU 비용 >> API 비용 (수백 배). "온프렘 이전기"는 추후 별도 프로젝트로 |
| 목소리 | **2차** | UI에 스피커 버튼 자리만 확보, API 응답 포맷에 `audioUrl` 필드 예약 |

## 페르소나 데이터 파이프라인 (가장 중요한 부분)

`me/` 볼트 전체를 절대 그대로 노출하지 않는다. **공개용 페르소나 문서를 별도 큐레이션**:

1. `persona/public_persona.md` 생성 — 볼트에서 **공개 가능 정보만** 추출 (프로필, 성격·가치관, 말투, 취향·영화·독서, 일의 정체성, 공개 가능한 경력 요약)
2. **제외 목록(하드 가드)**: 개인회생 등 민감정보(CLAUDE.md 절대 가드), 전화번호·주소·가족 실명, 일기·교환일기 원문, 알체라 내부 정보, 미취득 자격
3. 본인이 최종 검수 후 확정 → 이 문서만 시스템 프롬프트에 주입 (문서가 ~2만 토큰 이내면 벡터DB 없이 전체 주입 + 프롬프트 캐싱으로 충분·저렴 — RAG 인프라 과잉 방지)
4. 볼트 수정 시 재생성하는 스크립트(`persona/build.py` 또는 Node) 포함 — "데이터 파이프라인 설계"로 포트폴리오 어필 포인트

## 가드레일·QC 설계 (AI QA 경력 어필 핵심)

**구조적 방어 (가장 강력)**: LLM에는 DB 접근 도구가 아예 없음 — "DB 그대로 알려줘" 류 공격은 구조적으로 불가능. LLM이 아는 것 = 큐레이션된 페르소나 문서뿐.

레이어드 방어:
1. **입력 필터**: 길이 제한(500자), 언어 감지, 금칙 패턴(프롬프트 유출 시도, 역할 탈취 "모든 규칙 무시" 등) → 매칭 시 즉시 폴백 팝업 유도
2. **시스템 프롬프트 강화**: 페르소나 이탈 금지, 문서에 없는 사실 추측 금지("몰라요"→폴백 유도), 시스템 프롬프트 비공개 지시
3. **출력 검열**: 응답에서 민감 키워드·개인정보 패턴(전화번호·주민번호 정규식) 스캔 → 걸리면 폴백 응답으로 대체
4. **확신도 라우팅**: LLM 스스로 응답에 `"confidence": low` 태그 → low면 프론트에서 "이 질문은 본인이 직접 답해드릴게요" 팝업 자동 전환
5. **운영 방어**: IP당 분당 요청 제한, 일일 Claude API 지출 상한(초과 시 서비스 일시 잠금 + 본인 메일 알림), 대화 전량 로깅으로 사후 감사
6. **QA 시나리오 문서화**: 탈옥·주입 공격 테스트 케이스 30+ 작성·실행 결과를 `docs/qa-report.md`로 남김 — **AI QA 포트폴리오의 핵심 산출물**

## 폴백(직접 답변) 플로우

1. AI가 답변 불가 판단(확신도 low / 필터 트리거 / 사용자가 "직접 물어볼래요" 클릭)
2. 팝업: "이 질문엔 건우가 직접 답해드릴게요" + 자유텍스트 연락처 입력(카톡 오픈채팅/인스타/전화 등) + 개인정보 수집 동의 문구
3. 저장: 질문 + 직전 대화 이력 + 연락처 + 타임스탬프 → MongoDB `fallback_questions`
4. 알림: singunu17@gmail.com으로 즉시 메일(질문·대화맥락·연락처 포함)
5. (선택) 관리: 초기엔 Atlas UI + 메일로 충분, 추후 `/admin` 페이지

## UI/UX

기능이 단순한 만큼 디자인 완성도가 곧 첫인상. 접근:
- **Claude Code가 코드로 직접 하이엔드 디자인 구현** (클로드디자인 폴더의 가이드 철학 적용): 원페이지 몰입형 챗 UI, 다크/라이트, 타이핑 스트리밍 효과, Framer Motion 마이크로 인터랙션, 모바일 우선(인스타 유입 = 대부분 모바일)
- 첫 화면: 짧은 자기소개 히어로 + 추천 질문 칩 3~4개("어떤 일 하셨어요?", "영화 취향은?", "MBTI가?") → 바로 대화 시작
- 디자인 시안은 구현 전 **로컬 프로토타입(HTML)로 2~3안** 만들어 본인이 고르는 단계 포함 (Figma MCP 연동도 가능하나 코드 직행이 더 빠름)
- OG 이미지·파비콘·메타태그: 인스타/카톡 공유 미리보기 최적화 필수

## 사용자에게 제공한 보완 조언 (계획에 반영됨)

1. 원안의 Jenkins·S3+EC2 프론트 호스팅·온프레미스 LLM은 비용/효익상 제외 권고 → 수용됨
2. **가장 큰 리스크는 기술이 아니라 데이터**: 볼트에 민감정보가 많아 공개 페르소나 큐레이션+사람 검수가 필수 단계
3. 인스타 공개 = 불특정 트래픽: 비용 폭주 방지(레이트리밋+지출 상한 서킷브레이커)를 1차 릴리스에 포함
4. 폴백 연락처는 자유텍스트라 스팸 유입 가능 → 간단한 허니팟 필드 + 레이트리밋으로 방어
5. QA 시나리오 문서가 오히려 "AI QA 경력자" 포트폴리오의 차별점 — 산출물로 격상
